## Measurement checklist

You should measure the following before using the project in a resume, case study, or interview packet:

1. dataset size used
2. training time
3. inference latency (`p50`, `p95`)
4. key clustering metrics
5. baseline vs final improvement
6. optional operational metrics:
   - API startup time
   - low-confidence prediction rate
   - segment distribution
   - drift signals on fresh inference data

## 1) Dataset size used

Recruiters will often ask whether the project ran on a toy sample or a larger corpus. Even if the dataset is small, stating the exact size is better than hand-waving.

### How to measure it now


## 2) Training time

### Why measure it

Training time is a credible efficiency metric for a portfolio project, especially when the training pipeline is part of the story.

### How to measure it now

Use shell timing around the baseline training command:

```bash
mkdir -p docs/assets/metrics
/usr/bin/time -p -o docs/assets/metrics/train-time.txt \
  python3 train.py --input data/sample_customers.csv --artifacts_dir /tmp/customer_segment_measure
```

This records wall-clock training time in:

- `docs/assets/metrics/train-time.txt`

### Code locations to instrument

If you want training time stored in metadata automatically, instrument:

- `train.py` -> `main()`
  - add a `start_time` before reading data
  - compute `duration_seconds` after artifact save
  - include it in `save_metadata()`
- `train_v2.py` -> `run()`
  - this already computes duration in-memory; extend `save_artifacts()` to persist it in the main artifact metadata

### Recommended resume-safe phrasing

Only after capturing the timing artifact:

- "Baseline training completed in X seconds on the sample dataset"

## 3) Inference latency (`p50`, `p95`)

### Why measure it

Latency is one of the most recruiter-friendly proofs that the service is more than a notebook export.

### How to measure it now

Start the API first:

```bash
uvicorn api.main:app --host 0.0.0.0 --port 8000
```

In another terminal, run a simple percentile benchmark using only the standard library:

```bash
mkdir -p docs/assets/metrics
python3 - <<'PY' > docs/assets/metrics/predict-latency.json
import json
import time
import urllib.request
from statistics import median

url = "http://localhost:8000/predict"
payload = json.dumps({
    "age": 35,
    "annual_income": 75000,
    "spending_score": 62
}).encode("utf-8")

durations = []
for _ in range(200):
    req = urllib.request.Request(
        url,
        data=payload,
        headers={"Content-Type": "application/json"},
        method="POST",
    )
    start = time.perf_counter()
    with urllib.request.urlopen(req) as resp:
        resp.read()
    durations.append((time.perf_counter() - start) * 1000.0)

durations.sort()
def pct(values, p):
    idx = int(round((p / 100) * (len(values) - 1)))
    return values[idx]

result = {
    "n_requests": len(durations),
    "p50_ms": pct(durations, 50),
    "p95_ms": pct(durations, 95),
    "p99_ms": pct(durations, 99),
    "min_ms": min(durations),
    "max_ms": max(durations),
    "median_ms": median(durations),
}

print(json.dumps(result, indent=2))
PY
```

### Code locations to instrument

For permanent latency reporting, instrument:

- `api/main.py` -> `predict_segment()`
- `api/main.py` -> `ModelContainer.predict()`

Recommended additions:

- capture end-to-end request duration
- capture model-only prediction duration
- log segment ID and confidence to inspect latency by result class

### Recommended saved artifact

- `docs/assets/metrics/predict-latency.json`

## 4) Key model metrics

### Why measure them

For this repo, the right metrics are clustering metrics, not classification metrics such as AUC. This is unsupervised customer segmentation.

### What to measure for the current implementation

Baseline path:

- inertia
- silhouette score
- Davies-Bouldin score
- Calinski-Harabasz score
- cluster size distribution

Advanced path, once runnable:

- all of the above
- cluster balance
- problematic sample percentage
- separation ratio
- overall quality score
- business lift by segment

### How to measure them now

Baseline metrics are already saved in:

- `artifacts/model_metadata.json`

To extract them into a shareable table:

```bash
mkdir -p docs/assets/metrics
python3 - <<'PY' > docs/assets/metrics/baseline-metrics.csv
import csv
import json
from pathlib import Path

metadata = json.loads(Path("artifacts/model_metadata.json").read_text())
metrics = metadata["evaluation_metrics"]

rows = [
    ("n_samples", metadata["data_statistics"]["n_samples"]),
    ("n_clusters", metadata["hyperparameters"]["n_clusters"]),
    ("inertia", metrics["inertia"]),
    ("silhouette_score", metrics["silhouette_score"]),
    ("davies_bouldin_score", metrics["davies_bouldin_score"]),
    ("calinski_harabasz_score", metrics["calinski_harabasz_score"]),
    ("cluster_sizes", json.dumps(metrics["cluster_sizes"], sort_keys=True)),
]

writer = csv.writer(open("docs/assets/metrics/baseline-metrics.csv", "w", newline=""))
writer.writerow(["metric", "value"])
writer.writerows(rows)
PY
```

- `train.py` -> `evaluate_model()`

- `src/evaluation.py` -> `evaluate_internal_metrics()`

### Not applicable metrics

Do not use these unless you convert the project into a supervised ranking or classification problem:

- AUC

- Recall@K

Those are not native to the current clustering implementation.

## 5) Baseline vs final improvement

### Why measure it

This is the strongest story if you want to claim engineering impact:

- baseline = `train.py`
- final = `train_v2.py`

### Current blocker

This comparison is not yet runnable on the bundled sample data because:

- `train_v2.py` requires `customer_id`
- `data/sample_customers.csv` does not contain `customer_id`

So you should not claim a verified baseline-vs-v2 improvement until that data-contract issue is fixed.

### How to measure it after the fix

1. Create a schema-compatible input for v2
   - either add `customer_id` to `data/sample_customers.csv`
   - or modify `train_v2.py` to synthesize it during ingestion

2. Run both pipelines against the same logical dataset

```bash
python3 train.py --input data/sample_customers.csv --artifacts_dir /tmp/baseline_artifacts
python3 train_v2.py --input data/sample_customers_with_ids.csv --artifacts-dir /tmp/v2_artifacts --experiments-dir /tmp/v2_experiments
```

3. Compare:

- baseline:
  - `/tmp/baseline_artifacts/model_metadata.json`
- v2:
  - `/tmp/v2_artifacts/model_metadata_v2.json`

4. Build a comparison table

```bash
python3 - <<'PY' > docs/assets/metrics/baseline-vs-v2.md
import json
from pathlib import Path

baseline = json.loads(Path("/tmp/baseline_artifacts/model_metadata.json").read_text())
v2 = json.loads(Path("/tmp/v2_artifacts/model_metadata_v2.json").read_text())

b = baseline["evaluation_metrics"]
v = v2["evaluation_metrics"]["internal_metrics"]

rows = [
    ("silhouette_score", b["silhouette_score"], v["silhouette_score"]),
    ("davies_bouldin_score", b["davies_bouldin_score"], v["davies_bouldin_score"]),
    ("calinski_harabasz_score", b["calinski_harabasz_score"], v["calinski_harabasz_score"]),
]

print("| Metric | Baseline | V2 |")
print("|---|---:|---:|")
for metric, baseline_value, v2_value in rows:
    print(f"| {metric} | {baseline_value:.4f} | {v2_value:.4f} |")
PY
```

### Code locations

- `train.py` -> `evaluate_model()`
- `train_v2.py` -> `optimize_and_train()`
- `src/evaluation.py` -> `generate_evaluation_report()`
- `train_v2.py` -> `save_artifacts()`

## 6) Optional additional measurements worth capturing

### API startup time

Why:

- good operational signal for small inference services

How:

```bash
/usr/bin/time -p uvicorn api.main:app --host 0.0.0.0 --port 8000
```

Code locations:

- `api/main.py` -> startup event
- `ModelContainer.load_models()`

### Low-confidence prediction rate

Why:

- helps quantify how often the service is unsure

How:

- call `/predict` repeatedly across a representative input set
- collect `confidence_score`
- report percentage below a threshold such as `0.5`

Code locations:

- `api/main.py` -> `ModelContainer.predict()`
- `src/evaluation.py` -> `calculate_confidence_scores()`

### Segment distribution

Why:

- useful for detecting skew or drift after deployment

How:

- log `segment_id` from `/predict`
- aggregate counts over time

Code locations:

- `api/main.py` -> `predict_segment()`
- `train.py` -> `evaluate_model()` for baseline cluster sizes
- `src/evaluation.py` -> `evaluate_internal_metrics()`

### Drift score

Why:

- strong MLOps signal if you later wire inference logs into monitoring

How:

- compare new inference batches against training reference data using `DataDriftDetector.detect_drift()`

Code locations:

- `src/data_quality.py` -> `DataDriftDetector.detect_drift()`
- `src/data_quality.py` -> `_calculate_psi()`
